# Journal Entry 005 - 2026-02-28 - Fix UTC display bug

## Goal

Times set via `rem add --due` were displaying in UTC in the terminal instead of the user's local timezone. Reported as: "adding works fine in local time but displayed time is UTC".

## What Changed

Added `.Local()` before every `.Format()` call on time values used for display or export across 6 files:

- `internal/ui/output.go` — table view, plain list, rich detail, plain detail (7 call sites)
- `cmd/rem/commands/huh_helpers.go` — reminder label in interactive select/multi-select forms
- `cmd/rem/commands/update.go` — pre-populate due date field + change-detection comparison (3 call sites; all must use same format for string equality check)
- `internal/export/json.go` — `formatTimePtr()` helper used by `ToJSON()` and `ExportJSON()`
- `internal/export/csv.go` — CSV export of `due_date` and `remind_me_date` columns
- `cmd/rem/commands/import.go` — `--dry-run` preview output (2 identical call sites for CSV and JSON paths)

## Key Insights

- **Root cause**: EventKit returns `time.Time` values in UTC. Go's `time.Time` carries timezone info, but `.Format()` formats in whatever timezone the value is in — not local. Without `.Local()`, all displayed times were UTC.
- **`.Local()` is a no-op if already local** — safe to add unconditionally, no double-conversion risk.
- **update.go change-detection pattern**: The interactive update form pre-populates `dueStr` by formatting the existing due date, then compares `dueStr != origDueStr` after the form to detect changes. Both `dueStr` (pre-populate) and `origDueStr` (comparison baseline) must use the same format *and* the same timezone — so all three spots in update.go needed `.Local()` to stay consistent.
- **Export files**: Import side already used `time.ParseInLocation(..., time.Now().Location())` to parse into local time. Export side was writing UTC strings, breaking round-trips (export then re-import would shift times). Fixed by adding `.Local()` in export too.
- **Verified manually**: Added "tomorrow at 3pm" reminder from IST (UTC+5:30). Table showed `Mar 01, 15:00`, detail showed `Sun Mar 01, 2026 at 3:00 PM` — correct. Previously would have shown `09:30 UTC`.

## Decisions Made

- Export in local time (not UTC) to match import behavior and keep JSON/CSV human-readable with expected local times. This is consistent with how the import side parses (`ParseInLocation` with local).
- Did NOT add timezone annotation to display (e.g. "3:00 PM IST") — reminders don't have a stored timezone field like calendar events do, so the simpler approach (just `.Local()`) is correct. Dual-timezone display like in `go-cal` is not needed here.

## Key Insights

- **`baseURL: "/"` in Hugo breaks canonical URLs** — OG tags, sitemap, and canonical `<link>` all emit relative URLs like `/docs/commands/` instead of `https://rem.sidv.dev/docs/commands/`. Must be set to the full domain.
- **JSON-LD trailing comma bug**: The original structured data had a trailing comma before `"publisher"` inside a `{{ if not .IsHome }}` block — valid when that block rendered but the closing `{{ end }}` left a dangling comma when `IsHome` was true, producing invalid JSON. Fixed by restructuring into separate `{{ if .IsHome }}` / `{{ else if .IsPage }}` blocks.
- **llms.txt protocol**: Emerging standard at llmstxt.org. Place `llms.txt` at site root as a plain-text index for LLM crawlers. `llms-full.txt` for complete content dump. Add `<link rel="alternate" type="text/plain">` in `<head>` and reference in `robots.txt`.
- **`scroll-reveal` on container vs children**: Putting `scroll-reveal` on a wrapper div (opacity:0) means the entire section is invisible until JS fires — if JS is slow or IntersectionObserver threshold isn't met, section looks broken. Always put `scroll-reveal` on leaf/child elements, not layout wrappers.
- **`feature-card-highlight` pattern**: Simple modifier class that overrides border-color and bg to accent values — clean way to visually distinguish one card in a grid without duplicating the base card styles.
- **Agent chat mockup in pure HTML/CSS**: Used flex-column with `align-self: flex-end` for user bubble and `align-self: flex-start` for agent bubble. No JS needed. The `border-bottom-right-radius: 3px` / `border-bottom-left-radius: 3px` trick makes it look like a real chat app.

## Decisions Made

- **FAQPage JSON-LD on homepage only** — Not on doc pages. Doc pages get TechArticle + BreadcrumbList. FAQPage on every page would be semantically wrong and potentially penalized by Google.
- **`SoftwareApplication` schema instead of just `WebSite`** — rem is a software tool, not a website-as-product. `SoftwareApplication` with `operatingSystem: "macOS"`, `applicationCategory: "DeveloperApplication"`, and `featureList` array is more accurate and gives richer rich results.
- **Two chips not one sentence** — Marketing guidance (from growth-marketer agent): speed is the universal hook (concrete number), AI agents is secondary (narrower audience). Chips make both scannable at a glance without requiring reading.
- **Section order: features → agent demo → install → FAQ** — Per growth-marketer: put install CTA right after features while intent is high. FAQ at bottom is a trust-builder for people who scrolled all the way down.
- **Dropped "19 Commands" feature card** — Reads like a spec sheet, not a benefit. Count of commands is not a differentiator. Replaced with the AI agent card which IS a differentiator.
