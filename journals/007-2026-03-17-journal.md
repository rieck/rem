# Journal Entry 007 - 2026-03-17 - Natural date format support (#9)

## Goal

Implement issue #9 — support natural date formats like `21 Mar 2pm`, `march 21 at 2pm`, `2026-03-21 2pm` in the `--due` flag parser.

## What Changed

- `internal/parser/date.go`: Added `parseNaturalDate()` function (~70 lines) that handles day-month and month-day ordering, optional year, optional time, and the `at` keyword. Also added a `months` map for name→`time.Month` lookup (both abbreviated and full names).
- `internal/parser/date.go`: Extended `tryStandardFormats()` with 19 new format strings — ISO+bare-12h (`2006-01-02 3PM`), day-first with time variants (`2 Jan 2006 3:04PM`, `2 Jan 2006 15:04`), and full month day-first (`2 January 2006` variants).
- `internal/parser/date.go`: Wired `parseNaturalDate()` into `ParseDate()` fallback chain — placed after today/tomorrow/yesterday but before relative/weekday parsers.
- `internal/parser/date_test.go`: Added `TestParseDateNaturalFormats` with 16 test cases covering all formats from the issue, `at` keyword, year/no-year, day-first/month-first, and error cases.

## Key Insights

- **Go's time format system doesn't support bare hour+period** — `3PM` without minutes isn't a standard Go layout, so `2006-01-02 3PM` was added explicitly. The `parseNaturalDate` function delegates time parsing to the existing `parseTimeStr()` which handles bare hours via regex.
- **"at" normalization simplifies everything** — instead of adding separate regex patterns for `march 21 at 2pm` vs `march 21 2pm`, just stripping ` at ` from the input before parsing handles it cleanly.
- **Year ambiguity with month-first** — `mar 21 2026` could theoretically mean "March 21, 2026" or "March 2026 at 21:00". Resolved by checking if the number is a valid year (1000-9999) only after day+month are parsed. Since days max at 31, no collision.
- **Placement in fallback chain matters** — natural date parsing runs on lowercased input and must come before `parseTimeOnly()`, otherwise `Mar 21` would fail (not a time) and fall through to the generic error. Placed it early, right after the keyword handlers.

## Decisions Made

- Default to 9 AM when no time specified (consistent with existing `today`, `tomorrow`, `next week` defaults).
- No year validation beyond range check — if someone says `mar 21 1999`, it parses. Not worth restricting.
- PR opened as #10, closes #9.
