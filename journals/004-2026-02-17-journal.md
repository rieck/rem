# Journal Entry 004 - 2026-02-17 - Agent Skills & Background Update Check

## Goal

Port the agent skills distribution system and background update check from the `ical` project to `rem`. This gives users `rem skills install/uninstall/status` commands and automatic new-version notifications.

## What Changed

### New packages
- **`internal/skills/`** — Agent skill install/uninstall/status/detect logic. Ported from ical with `rem-cli` naming. Handles symlink replacement (dev setups), version tracking via `.rem-version` file, multi-agent support (Claude Code + Codex CLI).
- **`internal/update/`** — Background update check against GitHub releases API. 24h file cache at `~/.cache/rem/update-check`, 2s HTTP timeout, fire-and-forget goroutine pattern. Semver comparison with pre-release suffix stripping.
- **`skills.go`** (root) — `go:embed skills/rem-cli` directive exposing `EmbeddedSkills` as `embed.FS`.

### New CLI commands
- **`cmd/rem/commands/skills.go`** — `rem skills install/uninstall/status` with `--agent claude|codex|all` flag. Interactive `huh.NewMultiSelect` picker when no flag provided. Non-TTY auto-detects installed agents.

### Modified files
- **`cmd/rem/commands/root.go`** — Added `PersistentPreRun` (starts background update goroutine + NO_COLOR handling) and `PersistentPostRun` (prints update notice + skills staleness notice to stderr). `shouldCheckForUpdate()` skips for: `REM_NO_UPDATE_CHECK=1`, dev builds, json output, non-TTY, meta commands (version/completion/skills).
- **`scripts/install.sh`** + **`website/static/install`** — Post-install prompt for skill installation using `read -r answer < /dev/tty` for curl-pipe-bash compatibility.
- **Skill files** — Added skills commands to `SKILL.md` command table and `references/commands.md`. Added `REM_NO_UPDATE_CHECK` env var docs.
- **Deleted `skills/rem-cli/references/go-api.md`** — Stale since `pkg/client/` removal.
- **Website** — Added skills sections to `getting-started.md` and `commands.md`.
- **`README.md`** — Added AI Agent Skills section and updated architecture tree.
- **`CLAUDE.md`** — Added skills architecture docs, `charmbracelet/huh` library.

### New dependency
- `charmbracelet/huh` v0.8.0 — Interactive multi-select for agent picker.

## Key Insights

- **Direct port worked cleanly** — The ical skills/update packages were well-abstracted. Only needed to change string constants (`ical-cli` → `rem-cli`, `BRO3886/ical` → `BRO3886/rem`, `~/.cache/ical/` → `~/.cache/rem/`).
- **`versionStr` vs `Version`** — ical uses a `SetVersionInfo()` pattern with private `versionStr`. rem uses exported `Version` set directly from `main.go`. Skills and root commands just reference `Version` directly since they're in the same package. No need to add a `versionStr` alias.
- **`isInteractive()` name collision** — rem already had potential for naming conflicts in the commands package, so the skills helper was named `isInteractiveSkills()`.
- **`PersistentPreRun` addition** — rem's root command didn't previously have `PersistentPreRun`/`PersistentPostRun`. Added both. The `NO_COLOR` handling was moved into `PersistentPreRun` (previously only checked at flag level).
- **Update check is non-blocking** — Uses buffered channel + `select/default` in PostRun. If the goroutine hasn't finished by command completion, the notice is silently skipped. No user-visible latency.

## Decisions Made

- **Same agent targets as ical** — Claude Code (`~/.claude/skills/`) and Codex CLI (`~/.agents/skills/`). Matches the Agent Skills standard.
- **`REM_NO_UPDATE_CHECK`** env var (not `NO_UPDATE_CHECK`) to avoid conflicts with other tools.
- **Skills staleness notice** — If installed skill version doesn't match binary version, prints a yellow notice to stderr. Only shown once even if multiple agents have stale skills.
- **Deleted go-api.md** rather than updating it — the file pointed to the removed `pkg/client/` package. SKILL.md already links to go-eventkit directly.

## Session 2 — huh-based Interactive Mode for All Commands

## Goal

Replace all raw `bufio.NewReader` interactive prompts with polished `charmbracelet/huh` forms. Add `-i/--interactive` flag with multi-select batch operations to every command that benefits from it.

## What Changed

### New file
- **`cmd/rem/commands/huh_helpers.go`** — Shared helpers used across all interactive commands:
  - `huhTheme()` → `huh.ThemeCatppuccin()` (single source of truth, matches skills.go)
  - `isTTY()` → stdin terminal check (replaces `isInteractiveSkills()`)
  - `requireInteractive()` → returns error if not TTY
  - `reminderLabel(r)` → formats `"Title [List] (due: date) ⚑"` for selects, truncates at 50 chars
  - `reminderMultiSelect(title, reminders)` → huh MultiSelect returning selected IDs
  - `reminderSelect(title, reminders)` → huh Select for single reminder pick
  - `listSelect(title)` → fetches lists + huh Select for list name
  - `huhConfirm(title)` → huh Confirm returning `(bool, error)`

### Modified files (8 files, +1076/-190 lines)
- **`interactive.go`** — Complete rewrite. Main menu is now `huh.Select` loop with options: create, list, complete, delete, flag/unflag, manage lists, quit. Add form uses huh inputs/selects/confirm. Added sub-menus `runFlagMenuInteractive()` and `runListMgmtMenuInteractive()` that dispatch to per-command helpers. Removed all `bufio` usage.
- **`complete.go`** — Added `--interactive/-i`, `--list/-l`, `--flagged` flags. Custom Args validator allows 0 args when `-i`. `runCompleteInteractive(uncomplete bool)` fetches filtered reminders → multi-select → batch complete/uncomplete. Same helper shared by `uncompleteCmd`.
- **`delete.go`** — Added `--interactive/-i`, `--list/-l`, `--flagged` flags. Single-delete now uses `huhConfirm` when TTY (falls back to requiring `--force` for non-TTY). `runDeleteInteractive()` uses multi-select + confirm before batch delete.
- **`flag.go`** — Added `--interactive/-i`, `--list/-l` to both `flagCmd` and `unflagCmd`. `runFlagInteractive(listName)` fetches incomplete reminders → multi-select → batch flag. `runUnflagInteractive(listName)` fetches flagged reminders → multi-select → batch unflag.
- **`update.go`** — Implemented the already-declared `--interactive` flag. Args changed to `RangeArgs(0,1)` when `-i`. `runUpdateInteractive(idArg)`: if no ID, uses `reminderSelect()` picker; fetches current reminder; pre-populates all huh form fields; diffs against original to build `updates` map; only sends changed fields.
- **`lists.go`** — Added `-i` to `create`, `rename`, `delete` subcommands. `runListCreateInteractive()` uses huh Input. `runListRenameInteractive()` uses `listSelect()` + huh Input. `runListDeleteInteractive()` uses `listSelect()` + `huhConfirm()`. Single-delete also switched from bufio to `huhConfirm`.
- **`skills.go`** — Replaced `isInteractiveSkills()` call with `isTTY()`. Removed the now-dead `isInteractiveSkills()` function.

## Key Insights

- **huh form pre-population** — huh uses Go variable binding (`Value(&myVar)`), so setting the variable before `form.Run()` pre-populates the field. This made the update form straightforward — just initialize all bound vars from the fetched reminder.
- **Args validation pattern** — Using inline `func(cmd, args)` closures that check the interactive flag works well: `if interactive { return MaximumNArgs(0) }; return ExactArgs(1)`. This preserves strict validation for CLI usage while allowing no args in interactive mode.
- **Confirmation replacement** — For non-interactive single deletes, the old bufio `(y/N)` prompt was replaced with `huhConfirm` when TTY. For non-TTY (piped stdin), we now require `--force` instead of silently reading from pipe. This is safer.
- **ErrUserAborted handling** — Every `form.Run()` call checks for `huh.ErrUserAborted` (Ctrl+C) and prints "Cancelled." + returns nil. Consistent UX across all commands.
- **Interactive helpers accept filter params** — `runFlagInteractive(listName string)` and `runDeleteInteractive()` read from package-level flag vars. The interactive menu dispatchers pass empty strings. This allows both `rem flag -i --list Work` and the interactive menu to share the same code path.

## Decisions Made

- **Theme: `ThemeCatppuccin()`** — Already used in skills.go, keeps visual consistency.
- **Multi-select label format** — `"Title [List] (due: Jan 02, 15:04) ⚑"` with 50-char title truncation. Enough context to identify reminders without horizontal overflow.
- **Delete non-TTY behavior** — Changed from silently reading stdin (could accidentally delete) to requiring `--force`. Safer for scripts.
- **Update diff-based** — Only sends changed fields to `UpdateReminder()`, avoiding unnecessary AppleScript calls for flagged when it hasn't changed.
- **All interactive helpers are standalone functions** — Reusable from both `rem <cmd> -i` and the `rem interactive` menu. The menu is just a thin dispatcher.

## Session 3 — Extract Filter Construction into Testable Functions

## Goal

Prevent the `complete -i` bug (inverted boolean showing completed instead of incomplete reminders) from recurring by extracting inline filter construction into standalone, testable functions with full test coverage.

## What Changed

### New files
- **`cmd/rem/commands/filters.go`** — Four extracted filter functions:
  - `completeFilter(uncomplete, listName, flagged)` — builds `ListFilter` for complete/uncomplete interactive flow. `Completed` field set to `uncomplete` (false=show incomplete for completing, true=show completed for uncompleting). Flagged filter only applies on complete path.
  - `deleteFilter(listName, flagged)` — always shows incomplete, optional list/flagged filters.
  - `flagFilter(listName)` — shows incomplete, optional list filter.
  - `unflagFilter(listName)` — shows incomplete AND flagged, optional list filter.

- **`cmd/rem/commands/filters_test.go`** — 13 test cases covering all filter functions. Pure Go struct tests — no cgo, no EventKit, no TTY. Key tests:
  - `TestCompleteFilter/complete_shows_incomplete_reminders` — the exact assertion that would have caught the original bug
  - `TestCompleteFilter/uncomplete_shows_completed_reminders`
  - `TestCompleteFilter/flagged_ignored_on_uncomplete`

### Modified files
- **`complete.go`** — Replaced 14 lines of inline filter construction with `completeFilter()` call. Also fixed the original bug: `completed := !uncomplete` (wrong) was replaced by the extracted function using `completed := uncomplete` (correct). Removed unused `reminder` import.
- **`delete.go`** — Replaced 10 lines with `deleteFilter()` call. Removed unused `reminder` import.
- **`flag.go`** — Replaced inline filter logic in both `runFlagInteractive` and `runUnflagInteractive` with `flagFilter()`/`unflagFilter()` calls. Removed unused `reminder` import.

## Key Insights

- **The bug was a one-character fix** (`!uncomplete` → `uncomplete`) but the root cause was untestable inline code. Extracting pure functions that return `*ListFilter` structs makes the logic trivially testable without any mocking.
- **Filter functions are pure** — they take primitive args and return a struct. No service dependencies, no side effects. This is the ideal unit test target.
- **The `reminder` import moved** — all three command files no longer import `internal/reminder` directly; `filters.go` holds the sole import for filter construction. Cleaner separation.

## Decisions Made

- **Functions in a separate `filters.go` file** rather than co-located in each command file. Keeps filter logic centralized and the test file focused.
- **No mocking infrastructure** — tests just assert struct field values. No need for service interfaces or test doubles.
- **Filter functions are unexported** — package-internal helpers, not part of any public API.
